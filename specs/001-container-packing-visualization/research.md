# Research: 3D Container Packing Visualization

## Decision 1: Rendering stack
- **Decision**: Использовать `three` + `@react-three/fiber` + `@react-three/drei`.
- **Rationale**: Это зрелая и активно поддерживаемая связка для 3D в React/Next.js,
  позволяет быстро получить orbit controls и переиспользуемые примитивы сцены.
- **Alternatives considered**:
  - Чистый Three.js без React-обертки: больше boilerplate и сложнее интеграция в React UI.
  - Babylon.js: зрелый вариант, но менее нативный путь для React-ориентированного UI.

## Decision 2: Packing strategy
- **Decision**: Использовать детерминированный heuristic `first-fit decreasing` по
  контейнерам со слоистым размещением (floor/support aware) и строгими валидациями.
- **Rationale**: Требуется надежная, объяснимая и воспроизводимая логика для фиксированного
  набора данных, с приоритетом минимального числа контейнеров.
- **Alternatives considered**:
  - Полный оптимальный 3D bin packing (exact search): слишком высокая сложность для MVP.
  - Стохастические алгоритмы: хуже воспроизводимость, выше риск недетерминированности.

## Decision 3: Rotation constraints
- **Decision**: Разрешить только yaw-повороты по полу (0/90 градусов при прямоугольной
  проекции), без изменения вертикальной оси товара.
- **Rationale**: Полностью соответствует конституции и спецификации.
- **Alternatives considered**:
  - 6-позиционная 3D ротация: нарушает правило "не переворачивать на бок".

## Decision 4: Support validation
- **Decision**: Проверять опору каждого товара выше пола через контакт по площади:
  проекция по XY должна иметь ненулевое перекрытие с полом/верхними гранями нижележащих
  товаров на уровне `z`.
- **Rationale**: Явно исключает "висящие" товары и остается вычислимо простой.
- **Alternatives considered**:
  - Проверка только по центру массы: может пропустить частично висящие случаи.

## Decision 5: Dependency policy
- **Decision**: Использовать latest stable версии библиотек на всем протяжении разработки,
  без prerelease, и сверять интеграцию только с официальной документацией.
- **Rationale**: Прямое требование пользователя (FR-013, FR-014, FR-014a).
- **Alternatives considered**:
  - Фиксация версий на старте: более стабильна, но отвергнута явным решением пользователя.

## Decision 6: Contracts
- **Decision**: Зафиксировать JSON Schema для `PackingResult`.
- **Rationale**: Повышает проверяемость результата и снижает риск рассинхрона алгоритма и UI.
- **Alternatives considered**:
  - Только TypeScript-типы: меньше формализации для внешней валидации.
