# Feature Specification: 3D Container Packing Visualization

**Feature Branch**: `001-container-packing-visualization`  
**Created**: 2026-03-20  
**Status**: Draft  
**Input**: User description: "Реализовать визуализацию упаковки товаров в контейнер с минимизацией числа контейнеров, без висящих товаров, без переворота на бок, с использованием фиксированного примера заказа и контейнера; использовать только Next.js, готовые активно поддерживаемые решения, только последние версии библиотек и строго актуальную документацию."

## Clarifications

### Session 2026-03-20

- Q: Как трактовать правило "последние версии библиотек"? → A: Всегда ставить самые новые версии даже в процессе разработки.
- Q: Что делать при нехватке места в текущем контейнере? → A: Добавлять дополнительные контейнеры, показывать их списком в интерфейсе и заполнять по очереди, минимизируя общее количество контейнеров.
- Q: Какой источник считать "актуальной документацией"? → A: Только официальная документация библиотек.

### User Story 1 - Автоматическая укладка с минимальным числом контейнеров (Priority: P1)

Как пользователь логистического инструмента, я загружаю фиксированный набор товаров и
размер контейнера, запускаю расчет и получаю валидный план размещения всех единиц товара
в минимально возможном числе контейнеров.

**Why this priority**: это ключевая ценность решения: без корректного плана размещения
визуализация не имеет смысла.

**Independent Test**: можно проверить автономно запуском расчета на фиксированном заказе;
успехом считается полный план размещения без нарушений физики и с минимальным количеством
контейнеров для данного входа.

**Acceptance Scenarios**:

1. **Given** фиксированный заказ и внутренние размеры контейнера `12032x2352x2698`,
   **When** пользователь запускает расчет, **Then** система строит размещение с
   использованием минимального числа контейнеров.
2. **Given** построенный план, **When** проводится геометрическая проверка, **Then**
   ни один товар не выходит за границы контейнера и не пересекается с другими товарами.
3. **Given** построенный план, **When** проводится проверка опоры, **Then** каждый товар
   либо стоит на полу контейнера, либо имеет опору снизу и не "висит в воздухе".

---

### User Story 2 - Интерактивная 3D-визуализация размещения (Priority: P2)

Как пользователь, я открываю результат в браузере и могу вращать каждый контейнер, чтобы
визуально проверить расположение товаров и убедиться, что использованы все позиции заказа.

**Why this priority**: после расчета пользователю нужна наглядная проверка результата и
возможность визуального контроля качества укладки.

**Independent Test**: можно проверить автономно на заранее сохраненном плане размещения;
достаточно открыть страницу и убедиться, что контейнеры вращаются, а все товары видимы.

**Acceptance Scenarios**:

1. **Given** готовый план размещения, **When** пользователь открывает страницу результата,
   **Then** отображаются все использованные контейнеры с 3D-моделями всех размещенных товаров.
2. **Given** отображенный контейнер, **When** пользователь вращает сцену, **Then** модель
   корректно реагирует на взаимодействие и сохраняет точные позиции товаров.

---

### User Story 3 - Аудит ограничений и целостности входа/выхода (Priority: P3)

Как пользователь, я хочу получить проверяемый отчет по ограничениям (без учета веса), чтобы
доверять результату и видеть, что ни один товар не потерян и ни одно правило не нарушено.

**Why this priority**: прозрачность результата снижает риск ошибок в отгрузке и упрощает
проверку корректности алгоритма.

**Independent Test**: можно проверить автономно по отчету валидации для того же входа;
должны быть явно подтверждены ограничения и перечислены все размещенные единицы.

**Acceptance Scenarios**:

1. **Given** завершенный расчет, **When** формируется отчет проверки, **Then** отчет
   подтверждает отсутствие "висящих" товаров, выхода за границы и пересечений.
2. **Given** исходный список товаров с количеством, **When** формируется итог,
   **Then** каждая единица товара отражена в размещении без пропусков.

---

### Edge Cases

<!--
  ACTION REQUIRED: The content in this section represents placeholders.
  Fill them out with the right edge cases.
-->

- Одна или несколько единиц товара не могут быть размещены даже после добавления необходимых
  контейнеров (например, товар больше внутренних размеров контейнера): система завершает
  расчет с явной ошибкой и списком неразмещенных единиц.
- Товар помещается в контейнер только в одной допустимой ориентации по полу: система
  использует только поворот вокруг вертикальной оси, не переворачивая товар.
- Расчет находит несколько вариантов с одинаковым числом контейнеров: система применяет
  детерминированное правило выбора (повторяемый результат при одинаковом входе).
- Вход содержит поле `weight`: система игнорирует вес как фактор оптимизации и не влияет на
  раскладку из-за массы.
- Пользователь вращает контейнер в 3D: отображение не должно менять фактические координаты
  размещения и не должно "скрывать" товары.

### Functional Requirements

- **FR-001**: Система MUST принимать фиксированный входной набор товаров:
  - `1 x2 Force 1100 EFI (1300x2300x1260, weight 545.00)`
  - `8 x9 Force 720 (1192x2300x887, weight 443.00)`
  - `41 x9 RST 240 Ultra (570x1900x870, weight 167.00)`
  - `6336 x2 RST 501 Ultra (740x2170x1200, weight 207.00)`
  - `6352 x2 GP 601 Ultra (770x2170x1250, weight 240.00)`
  - `6360 x9 RR 240 Ultra (620x2120x870, weight 180.00)`
  - `31006 x1 Ace 1000 (1130x3150x1260, weight 417.00)`
- **FR-002**: Система MUST использовать внутренние размеры контейнера
  `12032x2352x2698` для каждого контейнера в расчете.
- **FR-003**: Система MUST размещать все единицы товара без пересечений и без выхода за
  границы контейнеров.
- **FR-004**: Система MUST запрещать размещение товаров "в воздухе"; каждый товар выше пола
  должен иметь опору снизу.
- **FR-005**: Система MUST разрешать только поворот товаров по полу (вокруг вертикальной оси)
  и MUST запрещать переворот на бок/вверх дном.
- **FR-006**: Система MUST оптимизировать раскладку по приоритету минимального числа
  контейнеров.
- **FR-007**: Система MUST строить браузерную 3D-визуализацию одного или нескольких
  контейнеров с возможностью вращения сцены пользователем.
- **FR-007a**: Система MUST автоматически добавлять следующий контейнер, если размещение в
  текущем контейнере невозможно без нарушения ограничений, и продолжать раскладку до
  размещения всех единиц или явной геометрической невозможности.
- **FR-007b**: Интерфейс MUST показывать список использованных контейнеров, где каждый
  контейнер визуализируется как отдельная 3D-сцена/вкладка и отражает порядок заполнения.
- **FR-008**: Система MUST визуализировать все размещенные единицы товара; скрытие или
  исключение "лишних" товаров запрещено.
- **FR-009**: Система MUST игнорировать вес товаров/контейнера при оптимизации и проверках.
- **FR-010**: Система MUST формировать проверяемый итог по размещению (количество контейнеров,
  список размещенных единиц, подтверждение валидности ограничений).
- **FR-011**: Система MUST быть реализована только как Next.js приложение, без использования
  других фронтенд-фреймворков в качестве основного рантайма.
- **FR-012**: Система MUST использовать преимущественно готовые и активно поддерживаемые
  решения с регулярными обновлениями и живым сообществом; кастомная реализация допустима
  только для доменной логики упаковки, которую невозможно покрыть зрелыми библиотеками.
- **FR-013**: Система MUST использовать только актуальные стабильные версии выбранных
  библиотек и MUST обновлять их до latest stable на протяжении разработки (не только в
  начале), исключая deprecated/archived зависимости.
- **FR-014**: Система MUST реализовывать интеграции строго в соответствии с актуальной
  официальной документацией используемых библиотек и фиксировать ссылки на эти источники в
  проектной документации.
- **FR-014a**: Источником истины для требований к интеграции MUST считаться только
  официальная документация соответствующих библиотек/платформ.

### Key Entities *(include if feature involves data)*

- **OrderItemType**: тип товара заказа; включает `id`, `name`, габариты (`width`, `length`,
  `height`), `weight`, и `quantity`.
- **OrderItemUnit**: конкретная единица товара после развертки `quantity`; ссылается на
  `OrderItemType` и имеет уникальный идентификатор единицы.
- **ContainerType**: параметры контейнера; для этой фичи фиксирован как
  `12032x2352x2698` (внутренние размеры).
- **ContainerInstance**: конкретный использованный контейнер с индексом и списком размещений.
- **Placement**: запись размещения единицы товара в контейнере; включает `containerIndex`,
  `itemUnitId`, координаты, ориентацию (только поворот по полу), габаритный прямоугольный
  объем в контейнерной системе координат.
- **PackingResult**: итог расчета; включает число контейнеров, список `Placement`,
  список неразмещенных единиц (пустой при успехе), и результаты валидаций ограничений.

## Assumptions

- Используется один тип контейнера с внутренними размерами `12032x2352x2698`.
- Входной набор товаров фиксирован данным заказом и является обязательным тестовым примером.
- Масса (`weight`) присутствует как атрибут входа, но не участвует в ограничениях и цели
  оптимизации.
- Цель оптимизации: минимальное число контейнеров; дополнительные критерии допустимы только
  как вторичный tie-breaker при одинаковом количестве контейнеров.
- Технологическое ограничение для реализации: только Next.js.
- Для недоменной функциональности (3D-рендер, UI-контролы, валидация, утилиты) приоритет
  отдается зрелым поддерживаемым библиотекам над самописными решениями.
- Под "последними версиями" понимаются последние стабильные релизы (без alpha/beta/rc), и
  они поддерживаются актуальными на протяжении всей разработки фичи.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Для фиксированного входного заказа 100% единиц товара размещаются либо система
  явно сообщает о неразмещаемых единицах (без скрытых потерь данных).
- **SC-002**: Для каждого успешного расчета подтверждаются 0 случаев пересечений,
  0 случаев выхода за границы контейнера, 0 "висящих" единиц.
- **SC-003**: Для одинакового входа система возвращает одинаковое количество контейнеров и
  идентичный план размещений в 100% повторных запусков.
- **SC-004**: Пользователь может вручную повернуть 3D-сцену каждого контейнера и визуально
  обнаружить все размещенные единицы товара без пропусков.
- **SC-005**: Основная метрика оптимизации выражена числом использованных контейнеров; решение
  считается корректным только если не найдено внутренне более компактное решение с меньшим
  числом контейнеров в рамках того же алгоритмического режима.
- **SC-006**: Для всех внешних библиотек, включенных в реализацию, 100% зависимостей имеют
  статус "активно поддерживается" и не имеют статуса archived/deprecated на момент фиксации.
- **SC-007**: Для 100% используемых сторонних библиотек в документации проекта указаны ссылки
  на актуальные официальные источники, по которым реализованы соответствующие части решения.
